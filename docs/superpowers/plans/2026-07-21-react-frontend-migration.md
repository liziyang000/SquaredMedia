# React 前台与 MacCMS 后台迁移实施计划

> **状态（2026-07-21）：** 本地前台功能迁移与自动化验收已完成。79/79 个旧模板已进入
> [迁移矩阵](../../react-template-migration-matrix.md)，并分别归入 React 页面/共享组件、MacCMS
> 播放器或 RSS 后端直通、或明确 `410` 退场。`apps/web` 已实现发现、内容/播放、账号、互动和
> 系统页面族；`server/react-api.php` 提供带字段白名单、session 与 CSRF 的本地验收契约。
> `pingfangapi` 生产插件代码、独立归档和部署安装链已经建立，并已通过 API-only 发布到生产。
> 真实 MacCMS 内容读取已验证 227,572 条服务端分页、分类总数、详情、搜索和首页。最新门禁已通过 57 个 React
> 单元/契约测试、PHP/仓库测试、5 个 Playwright 场景、六个响应式宽度、类型检查、生产构建、
> 模板兼容和发布包验证。真实 MacCMS 会话/播放器验收、生产 Nginx/`301`、React 独立
> 发布、真实播放器线路和灰度切流尚未实施；路由懒加载、自动无障碍扫描、
> 视觉回归和旧预览清理属于后续非阻断质量工作。因此当前结论是“本地功能迁移和验收完成”，
> 不是“生产迁移完成”。本地公开前台使用 React、MacCMS 继续作为后台、采用不包含
> `index.php` 的新 URL，并且不开展 SEO 工作；旧页面地址仅为兼容历史链接、收藏和分享记录而
> 一对一 `301` 到新 URL。

**目标：** 将当前 `pingfangvideo` 的全部公开页面和交互迁移到 React，同时保留 MacCMS 作为内容、用户、采集、插件、数据库及播放器策略后台。

**推荐架构：** 使用 Vite 构建 React 单页应用（SPA），使用 React Router 管理前台路由；Node.js 仅用于本地开发、CI 和生产构建，不在生产环境常驻。构建产物由宝塔/Nginx 直接托管，浏览器通过同域名下经过验证的 MacCMS 接口或最小兼容插件读取和写入数据。

**URL 策略：** 新 URL 是 React 站内导航和对外分享的唯一页面地址；现有 MacCMS 页面 URL 一对一返回 `301` 到对应新 URL。`/api.php/**`、后台入口、上传文件和必要的 MacCMS 静态资源继续由 PHP/Nginx 处理，不能被 SPA 回退规则接管。

**技术栈：** React、Vite、React Router、TanStack Query、React Hook Form、Zod、TypeScript、Vitest、Testing Library、MacCMS V10/ThinkPHP、PHP 8.4、Node.js 22.22+（仅构建）、现有 CSS/品牌资源、Playwright、现有 Node/PHP 测试与 Bash 发布脚本。

## 不可妥协的成功标准

- `template/pingfangvideo/html/**` 当前 79 个模板或片段全部进入迁移矩阵，每项都有 React、MacCMS 后端直通或明确退场结论。
- 首页、列表、搜索、详情、播放、会员、设备、评论和反馈均有明确的新入口与自动化验证。
- MacCMS 后台、采集、分类、用户、设备会话、插件任务和数据库工作流保持可用。
- 未登录用户不能通过新接口、旧 URL 或播放器桥接端点绕过播放及会员权限。
- 新 URL 可直接访问和刷新；Nginx 能返回 React 入口，React 能恢复正确页面和数据状态。
- 旧页面 URL 均为单跳、一对一 `301`；不存在重定向链、循环、批量跳首页或参数丢失。
- Sitemap 等纯 SEO 输出不迁移；RSS 若确认仍有真实消费者，则继续由 MacCMS 后端直通，不由 React 重建。
- 任何阶段都能切回已验证的旧 MacCMS 主题和上一版 React 静态产物。
- 完成不能只依据 HTTP 200；必须同时验证页面内容、登录态、实际播放器、源站、CDN 和浏览器缓存。

## 新 URL 草案

最终规则必须以第一阶段生成的现网 URL 清单为准。下表先固定 URL 设计方向，避免实现期间各模块自行命名。

| 页面      | 旧 URL 示例                                      | 新 URL                     | 迁移行为                         |
| --------- | ------------------------------------------------ | -------------------------- | -------------------------------- |
| 首页      | `/`                                              | `/`                        | React 直接响应                   |
| 影片库    | `/index.php/vod/show.html`                       | `/videos`                  | 旧地址 `301`                     |
| 分类      | `/index.php/vod/type/id/42.html`                 | `/category/42`             | 旧地址 `301`，保留有效筛选参数   |
| 搜索      | `/index.php/vod/search/wd/关键词.html`           | `/search?wd=关键词`        | 旧地址 `301`，关键词正确编码     |
| 影片详情  | `/index.php/vod/detail/id/367756.html`           | `/vod/367756`              | 旧地址 `301`                     |
| 播放      | `/index.php/vod/play/id/367756/sid/1/nid/2.html` | `/watch/367756/1/2`        | 旧地址 `301`，不得丢失线路和集数 |
| 用户中心  | `/index.php/user/index.html`                     | `/account`                 | 未登录时转到 `/login`            |
| 收藏      | `/index.php/user/favs.html`                      | `/account/favorites`       | 保留会话和分页                   |
| 播放记录  | `/index.php/user/plays.html`                     | `/account/history`         | 保留会话和分页                   |
| 登录设备  | 当前 `pingfangdevice` 页面                       | `/account/devices`         | React 页面调用原插件能力         |
| 留言/报错 | 当前 gbook/book 路由                             | `/feedback`、`/report`     | 旧地址分别 `301`                 |
| 兼容模块  | 当前 art/topic/actor 等路由                      | 按迁移矩阵确定的 React URL | 逐条映射，不允许通配跳首页       |

Sitemap 等 SEO 路由不创建 React 替代地址；现有相关端点应在切流前明确返回 `404` 或 `410`。RSS 不属于 React 页面：只有确认现网仍有消费者时才保留 MacCMS 原端点，否则单独评审退场。

## Nginx 路由优先级

路由必须按以下顺序匹配，避免 SPA 回退吞掉后台或接口：

1. 精确匹配旧页面规则并返回一对一 `301`。
2. MacCMS 后台、`/api.php/**`、插件接口、`/upload/**`、必要 PHP 和静态资源按原链路处理。
3. React 带内容哈希的静态资源直接返回，允许长期缓存。
4. 已知 React 页面路由使用 `try_files $uri /index.html` 回退。
5. 其余未知路径返回 `404`，不能统一进入首页或后台。

## 验证命令

迁移期间始终保留当前门禁：

```bash
npm test
npm run lint
npm run lint:template
npm run verify:compat
npm run verify:preview
npm run package
npm run verify:release
```

当前 React 基座已固定并由 CI 执行：

```bash
npm run lint:web
npm run typecheck:web
npm run test:web
npm run build:web
npx playwright install chromium
npm run test:e2e
```

独立 React 发布链建立后再增加：

```bash
npm run package:web
npm run verify:release:web
```

发布完成后的验证至少包括：

- 新页面直达刷新、React 内容、文档标题、加载状态和错误状态。
- 旧页面到新页面的单跳 `301`、参数保留和无循环。
- MacCMS API、后台、上传资源不被 SPA 回退规则截获。
- 匿名、登录、过期、撤销设备及无权限状态。
- 搜索、筛选、分页、收藏、历史、评论、留言和报错写入。
- 真实播放器 iframe/媒体请求、线路切换、上下集和播放记录。
- 源站、CDN、浏览器磁盘缓存与实际运行版本一致。

## 主要风险与约束

- 仓库不包含完整 MacCMS 核心，线上接口能力必须通过真实版本探针确认。
- 官方 Provide 接口可能返回超出前台需要的字段，不能原样暴露给浏览器。
- 登录和设备管理依赖 MacCMS Cookie、插件 hook 与同源 Ajax，错误代理会造成“登录成功但仍是游客”。
- 当前播放器依赖 MacCMS 生成的运行时数据和脚本，不能仅把模板 HTML 改写成 React JSX。
- SPA 依赖浏览器 JavaScript；入口或分包资源加载失败会直接影响页面可用性，必须提供监控、重试和版本回退。
- Nginx 回退顺序错误可能截获后台、接口、上传文件或旧 URL 重定向，必须用路由矩阵逐条测试。
- CDN 新鲜不代表浏览器磁盘缓存已更新；验证必须查看实际加载资源和播放器 iframe。
- 静态预览、PHP 预览和生产模板目前各自维护渲染逻辑，删除前必须由 React fixture/E2E 完整接管。

## 实施前剩余确认

- [ ] 首期接受“React 页面 + MacCMS 原生播放器桥接”，不同时启动自研播放器项目。
- [ ] staging 使用与生产一致的 MacCMS 版本、播放器配置、Cookie 域和 Nginx 路由规则。
- [ ] RSS 是否仍有真实消费者；若无，则与相关旧端点一起单独退场。

## Definition of Done

- 所有迁移矩阵项目已完成，无未解释的旧模板入口。
- 所有 React 站内链接使用新的无 `index.php` URL。
- 所有旧页面 URL 一对一单跳 `301` 到正确的新 URL；无替代页面的非页面端点有明确退场结论。
- 计划和实现不包含 SEO 生成、优化、提交或排名验收工作。
- React 页面、MacCMS 后台、用户会话、插件、播放和内容更新链路均通过自动化及真实环境验收。
- CI 同时覆盖 React、MacCMS 兼容插件、发布包、Nginx 路由和重定向规则。
- 灰度观察期达到预先确定的 SLO，且已成功演练回滚。
- 旧主题只作为经过验证的回滚资产保留，不再承担公开页面主流量。
